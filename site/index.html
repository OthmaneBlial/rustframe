Rust / Tao / Wry / SQLite

Build local-first desktop workflow tools without starting from a full native project.

RustFrame keeps the app folder plain, lets the runtime own the native shell, provisions embedded SQLite, and exposes filesystem or shell access only when the manifest and trust model explicitly allow it.

Core Commands

phase 1

Run the flagship workflow now. Scaffold or package the same runtime path later.

cargo run -p rustframe-cli -- new archive-reviewer
cargo run -p rustframe-cli -- dev research-desk
cargo run -p rustframe-cli -- package research-desk
cargo run -p rustframe-cli -- export research-desk
cargo run -p rustframe-cli -- eject research-desk
Packaged app path

Host-native bundles already sit inside the workflow, not outside it.

Linux dist/linux/<app-id>-<version>-linux-<arch>/
Windows dist/windows/<app-id>-<version>-windows-<arch>/
macOS dist/macos/<app-id>-<version>-macos-<arch>/

Declared relative filesystem roots are copied into these outputs, so a workflow app can ship its local workspace and helper tools with the bundle.

Fit

RustFrame is for a narrower job than “desktop apps in general.”

The strongest fit is a local-first workflow tool with mostly frontend UI, structured local data, and a small amount of explicit machine access.

Use RustFrame when

You want the runtime to own the desktop shell around a frontend-heavy app.

  • The app should stay mostly HTML, CSS, and JavaScript.
  • You want embedded SQLite without building the full desktop stack yourself.
  • You need scoped filesystem access, allowlisted commands, or multi-window views.
  • You want to start small and eject only if the app outgrows the hidden runner.
Do not use RustFrame when

You already know the product needs broad native depth or a larger ecosystem.

  • The app works fine as a browser tab or PWA.
  • You need mature plugin ecosystems and deeper native APIs immediately.
  • You want Chromium-level rendering consistency across every host.
  • You are already productive in Tauri or Electron and not feeling friction.

Target Workflow

One concrete job RustFrame should own: a local research desk.

research-desk is the current proof point: a workflow that is awkward in a browser tab, but still does not need a full native-first architecture on day one.

01

Scope a local archive

Declare workspace/ and tools/ in the manifest so the app works against one known archive instead of the whole machine.

02

Index source files into SQLite

Store document metadata, status, reviewer notes, and queue state in the runtime-managed local database.

03

Open focused reader windows

Use secondary windows for deep reading while keeping one runtime, one bridge contract, and one shared local dataset.

04

Run a hardened Python indexer

Allowlist a local indexing command and capture stdout or stderr directly in the UI without exposing a general shell.

05

Export queues or package the tool

Ship the same workflow as a host-native desktop bundle once it stops being an experiment.

Manifest snapshot

The workflow stays explicit in one contract.

{
  "appId": "research-desk",
  "security": { "model": "local-first" },
  "filesystem": {
    "roots": ["workspace", "tools"]
  },
  "shell": {
    "commands": [
      {
        "name": "indexWorkspacePy3",
        "program": "python3",
        "args": ["index_workspace.py", "../workspace"],
        "cwd": "tools"
      }
    ]
  }
}

This is the pitch in practice: a frontend-first app with local data, scoped file access, and one explicit automation path. The same relative roots are copied into export and package outputs.

Ship

Packaging is already part of the flagship app story.

RustFrame should matter when the workflow stops being a demo and starts needing a real desktop install path, even if the runtime is still intentionally small.

Package command

Build on the native host.

cargo run -p rustframe-cli -- package research-desk

The generated or ejected runner becomes a host-native bundle without exposing a full native project up front.

Bundle outputs

What the packaging path already produces.

  • Linux portable bundle directory, shell install scripts, and archive.
  • Windows portable directory, PowerShell install scripts, shortcuts, and zip.
  • macOS app bundle, shell install scripts, and archive.
  • Declared relative filesystem roots copied with the app, so bundled workspaces survive release builds.
Current limits

Be honest about what is still early.

  • Cross-host validation still needs the matching native host toolchain.
  • Signing, notarization, and update channels still need more production polish.
  • Linux still has the heaviest GTK, WebKitGTK, and display-stack constraints.
